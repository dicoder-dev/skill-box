## 4. 问题与方案

- **规则冲突**: README 旧版"任务结束"段说"任务结束后还继续追加内容(开新文件)"是"不要",新版要确保不会让人误读成"任务结束后不许追加对话轮次"。解决方案:把"对话轮次追加"显式写在"任务中"段,跟"任务结束"段的"开新文件"区分清楚(同一任务内继续追加 ≠ 任务结束还硬塞新话题)。
- **粒度平衡**: 用户的诉求是"每轮对话都维护",但每条 1-2 行的小对话都建独立 task 文件会很碎。解决方案:延续现有"一个目标 = 一份文件"原则,**文件内的每轮对话都追加**,而不是每个对话建一份文件。
- **触发点**: 不能依赖 git commit 触发维护(很多对话不会有 commit,比如纯讨论 / 澄清)。把触发点改成"每轮对话结束后追加",写在规则里。
- **示范先行**: 改完规则立刻在本任务文件里追加一段 `## 1.1 对话轮次 (HH:MM)`,作为示范,后续对话有据可依。
