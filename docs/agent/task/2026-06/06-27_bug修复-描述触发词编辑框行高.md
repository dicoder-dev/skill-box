## 4. 问题与方案
> 现象:用户点"编辑"后,主页"描述"输入框显示高度约 100px+,"触发词"约 100px+,
>       远超 rows=2/rows=1 对应的 57px/38px。

- **定位过程:**
  1. 看 git log 最近三次提交都是 textarea 行高锁定相关(c07e929/11d9b51),以为是新引入的回归
  2. 读 .desc-editor / .triggers-editor CSS,发现已经用 `height + min-height` 精确锁
  3. 翻 style.css 全局,翻 .editor-field-full 局部,定位到第 2839 行:
     `.editor-field-full textarea { min-height: 100px }` —— 单类选择器与 .desc-editor 同优先级,
     后写的 `.desc-editor { min-height: 57px }` 胜出,但 100px 是写在 **App.vue scoped 之前**
     的全局,特异性对比错了
  4. 实际上:scoped CSS 编译时会加 `[data-v-xxx]`,而 `.editor-field-full` 是子组件根类
     透传,scoped 不会影响它的特异性。两者同样是 (0,1,0) 单类选择器,**位置靠后者胜出**。
     看 .desc-editor 在第 2152 行(更早),.editor-field-full 在 2839 行(更晚) → 后者胜出,
     把 min-height 拉到 100px(但 height:57px 在没有 min-height 限制时本应生效,
     加上 min-height:100px 后,min-height 优先生效,实际高度 ≥ 100px)

- **方案:** 用 `:not(.desc-editor):not(.triggers-editor)` 把单行/双行精确行高的编辑器
  从全局 min-height 100px 规则中排除,只让 body 等大段内容走 100px 最小高度。
  - 选择器特异性: (0,2,1) → 高于原来 (0,1,1)
  - 编辑器侧: `min-height: 57/38px` 单独生效,精确锁行高
  - body 等其他大段内容: 100px 最小高度保持不变

- **教训:**
  1. `min-height` 优先级高于 `height`,写 `height: 57px` 但没显式覆盖 `min-height` 时
     容易被外层全局 `min-height` 拉高
  2. App.vue 用 scoped 不会自动给子组件根类加 `[data-v-xxx]`,全局规则能覆盖子组件 scoped 规则,
     排查时不能只看 scoped 内的冲突,也要看全局 / 父级同名类
  3. 命名上:`.editor-field-full` 这种"通用"类容易撞上其他组件的 textarea,
     后续新编辑器尽量带具体语义类(.desc-editor/.triggers-editor/.body-editor)
