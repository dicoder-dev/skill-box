# docs/agent/task — 任务过程文件

> 每一个"有具体目标"的对话 / 子任务都应当在这里留一份过程文件。
> 不是给人看的进度汇报,是给 **未来的 Claude / 你自己** 回看的"我们当时怎么做的"。

## 何时建

**默认都要建。** 任何对话都值得留一份过程文件 —— 哪怕只是闲聊,也建一份简短记录,
保留当时的上下文、用户原话、Claude 答复,日后翻看时能记起"为什么这样答"。

- 用户给了一个具体任务(不是闲聊) → 完整过程文件
- 一两句话就能答完的闲聊 → 简短记录(需求 / 答复 / 决定)
- 纯信息查询("XX 文件在哪") → 可以省略

不建的话,几个月后回看会完全想不起来当时聊过什么。

## 同一对话里"多个需求"怎么拆文档

> 这是**最常踩的坑**。一个对话里经常会出现"做着做着,用户又加塞一个完全不相关的事",
> 这种时候要果断**新开一份 task 文档**,不要硬塞进当前那份。

### 判定原则:**按"需求主题"拆,不是按"对话"拆**

| 场景 | 怎么拆 | 为什么 |
|---|---|---|
| 同一对话内,做的还是最初那个需求(只是澄清 / 细化 / 修 bug) | **更新当前 task 文档**(在原文档里追加"对话轮次"和"需求回流") | 主题没变,过程是一体的 |
| 同一对话内,用户提了一个**新的、不相关**的需求(比如"再帮我看看 X") | **新开一份 task 文档**(`MM-DD_<任务类型>_<新主题>.md`) | 主题不同,过程要分开,事后好回查 |
| 同一对话内,做的还是同一需求但**目标/范围明显扩展**(比如阶段 1 → 阶段 2) | 看情况:延续同一份(在"需求"里追加阶段说明) 或 新开一份(当跨度大到要分两个里程碑) | 视规模定 |

### 判定口诀

- **"还在做这件事"** → 续写当前文档
- **"开始做另一件事"** → 新开文档,文件名用新需求的主题
- **拿不准** → 新开一份,而不是把两件事混进同一份

### 反模式(❌ 不要这么干)

- 一个对话里干了两件不相关的事,只建了一份 task 文档 → 几周后翻看根本分不清哪段是哪件
- 把新需求塞进"需求回流"小节,却真的去实现了 → "需求回流"是临时记录,不是任务主体
- 在原 task 文档里"借壳"写新需求,只是因为懒得建新文件 → 失去按主题检索的能力

## 目录结构(按月份分组)

```
docs/agent/task/
├── README.md               # 本文件
└── YYYY-MM/                # 按月份分子目录,一个月一个文件夹
    ├── MM-DD_<任务类型>_<主题>.md
    ├── MM-DD_任务类型<主题>.md
    └── ...
```

- **月份目录命名:`YYYY-MM/`**(例如 `2026-06/`)
- **文件名:`MM-DD_<任务类型>_<主题>.md`**
- 同月份的任务都进同一个 `YYYY-MM/` 子目录,避免 `task/` 根目录文件膨胀
- 新月份第一天如果发现还没有对应目录,先 `mkdir -p` 再放文件

## 文件命名

```
docs/agent/task/YYYY-MM/<任务类型>_<主题>.md
```

- **月份目录:`YYYY-MM/`**(例如 `2026-06/`),按对话开始的月份归类
- **文件名:`YYYY-MM-DD_<主题>.md`**,日期用对话开始的日期(本地时区)
- 主题:用动名词短语,简洁可搜索
  - `docs/agent/task/2026-06/06-23_功能开发_claude-分层协作方案.md` ✅
  - `docs/agent/task/2026-06/06-23_bug修复-sys_user-搜索-参数校验.md` ✅

## 文件结构

```markdown
# <任务主题>

**日期:** YYYY-MM-DD
**状态:** 进行中 | 已完成 | 已暂停

## 1. 需求
用户原始诉求 + 任何澄清后的细化目标。

## 2. 任务列表
- [ ] 子任务 1
- [ ] 子任务 2
- [x] 已完成子任务

## 3. 执行进度
时间倒序记:
- HH:MM 完成 XXX
- HH:MM 发现 XXX 问题
- HH:MM 决定 XXX

## 4. 问题与方案
> 开发中遇到的非平凡问题(>5 分钟定位或设计取舍)。
> 每个问题:现象 → 定位 → 方案 → 教训

## 5. 需求回流
> 用户临时加塞 / 计划外需求。完成后并入 `docs/project/需求规划.md`。

## 6. 测试报告
> **仅"功能开发"与"问题修复"类任务必填。**
> 纯文档 / 纯讨论 / 闲聊类任务可省略此节,但要在文件结构里写明"本任务不涉及自测"。
> AI 必须**先自测**,把测试报告写进 task 文档,再向用户报告完成。

## 7. 总结(任务结束时填)
- 完成了什么
- 留下了什么(代码 / 文档 / 决策)
- 留给下次的事
- 复盘:哪里做得好,哪里能改进

## 8. 改动的文件
> **AI 必须维护,逐轮更新。** 任何 task 涉及到了文件改动(新增 / 修改 / 删除),
> 都要在这里列清楚,便于事后 `grep` 追溯"这个改动落在哪几个文件里"。
>
> **何时写:**
> - 任务开始就预留此节;第一行写 `- 暂无(任务进行中)`
> - 每一轮对话结束、产生了文件改动 → **立即追加**到本节(不要等任务结束再补)
> - 任务结束(填总结时)→ 复核一遍本节,删除/调整不准确条目,确保与 git diff / commit 完全一致
>
> **写法规范:** 按"变更类型"分组,**完整仓库相对路径**(`api-server/...` / `frontend/...` / `docs/...`),
> 每条带一句话说明作用。**不要写完整代码**,只列路径 + 说明。

```markdown
## 8. 改动的文件

### 8.1 新增
- `api-server/internal/gapi/controller/skillbox/cskill/scope_status.a.go` — scope 两级状态查询接口
- `frontend/src/api/skillbox/skills.js` — 加 `getSkillScopeStatus` 客户端

### 8.2 修改
- `frontend/src/views/SkillsView.vue` — scope 区从单级多选改两级(工具 × 作用域)
- `frontend/src/core/i18n/zh-CN.js` — 加 `scopeToolsRow` / `scopeTargetsRow` / `scopeEmpty` / `scopeHitCount`
- `frontend/src/core/i18n/en-US.js` — 同步英文 i18n key

### 8.3 删除
- `frontend/src/views/SkillsView.vue` — 删旧 `toggleScope` / `isScopeActive` / `projectLabel` / `activeScopes`(如属于修改行内的清理,可合并到 8.2 一行,无需单列)
```

### 8.4 不涉及文件改动的任务

纯讨论 / 纯澄清需求 / 纯查文档 类任务,本节直接写:

```markdown
## 8. 改动的文件

> 本任务为纯讨论 / 纯澄清,未涉及任何文件改动。
```
```

## 维护规则

### 新建任务

- **任务开始**:复制本目录下结构(`# 任务主题` / `**日期:**` / `**状态:**` / `## 1. 需求` / `## 2. 任务列表`),
  改名放到当月目录 `YYYY-MM/`(没有就 mkdir),立即填"需求 / 任务列表"
- **同一对话内出现新需求**:
  - 先按上文"同一对话里'多个需求'怎么拆文档"判定
  - 仍属当前需求 → 在原文档里追加"对话轮次"和(或)"需求回流"
  - **新需求 → 立即新开一份 task 文档**(`MM-DD_<任务类型>_<新主题>.md`),不要混进当前文档
  - 在当前文档的"对话轮次"里加一句"另起新任务:见 `<新文件>`"
- **任务中**:
  - 每次有进展,更新"任务列表"和"执行进度"
  - 凡是产生了文件改动,同步更新 `## 8. 改动的文件`(按"新增/修改/删除"分类,带完整路径 + 一句话说明)
  - **每轮对话结束后必须追加一段** `## N.M 对话轮次 (HH:MM)`(见下方格式),
    即便本轮对话没有新代码 / 没有 git commit 也要追加
- **功能开发 / 问题修复类任务**:
  - AI 必须**先自测**(单元测试 + 必要的接口/端到端校验),把结果写到 `## 6. 测试报告`
  - 没跑过测试就向用户报告"完成" → 视为偷工,需要回补
  - 自测失败要先修,直到通过才能进入"任务结束"流程
- **任务结束**:
  1. 把状态改为 `已完成`
  2. 填"测试报告"(功能开发 / 修复类必填,其他类型可写"不涉及")
  3. 填"总结"
  4. **复核 `## 8. 改动的文件`** — 必须与本次任务实际改动一致(可用 `git diff --name-only HEAD~N..HEAD` 校验),漏列 / 多列都不允许
  5. 决定是否要更新 `docs/project/` 业务文档或 `docs/agent/memory/` 长期记忆

### 对话轮次追加格式

每轮对话结束后,在该任务的 task 文件里追加一段,固定结构:

```markdown
## N.M 对话轮次 (HH:MM)

> 用户原话(关键句摘抄,不是逐字)

- **本轮做了:** 完成的修改 / 决定(带文件路径)
- **本轮决定:** 重要取舍、设计选择
- **本轮待办:** 留给下一轮的事
- **状态更新:** 任务列表勾选变更 + 状态字段
```

- `N` 是任务序号(1, 2, 3 ...),`M` 是该任务内第几轮对话,从 1 起
- 时间用对话开始的本地时间
- 没有新代码的对话(纯讨论 / 澄清需求)也要追加,记录"用户原话 + 决定"
- 一份 task 文件内的所有对话轮次,按时间正序追加(不要倒序),便于翻阅时从头读到尾
- 本轮若产生了文件改动,**对话轮次小节里要列出关键路径**,并把完整清单同步到 `## 8. 改动的文件`

## 测试报告填写规范

> 适用于"功能开发"和"问题修复"两类任务。其他类型(纯文档/纯讨论/重构等)无需自测,
> 但需要在文件结构里把 `## 6. 测试报告` 留作小节,内容写"本任务不涉及自测"。

### 何时必须自测

- 新增/修改了 **任何** 后端 Go 代码 → `go test ./...` 必须通过
- 新增/修改了 **任何** 前端 Vue/TS 代码 → `npm run build`(及必要的 `npm run lint`/`npm run test`)必须通过
- 涉及 **接口行为变更**(controller/service/路由/参数) → 必须用真实请求验证(curl / 单元测试 / 接口测试)
- 涉及 **DB schema 变更** → 必须在本地库跑一次迁移 / 启动一次服务确认无 panic
- 涉及 **配置/环境变量** → 必须跑一次启动,确认读取正确
- 涉及 **文件读写 / 路径 / 权限** → 至少跑一次真实落盘验证

### 测试报告必填内容

```markdown
## 6. 测试报告

**自测时间:** YYYY-MM-DD HH:MM
**自测人:** AI(本轮 Claude)
**自测范围:** <本次改动涉及的范围,如 sskill service + skillstore>

### 6.1 自动化测试
- `go test ./...` 结果: ✅ 通过(耗时 Xs)/ ❌ 失败
- `go vet ./...` 结果: ✅ 通过
- 前端 `npm run build` 结果: ✅ 通过 / ❌ 失败
- 前端 `npm run lint` 结果(若改前端): ✅ 通过 / ❌ 失败

### 6.2 手工 / 接口验证
- [x] 用例 1: <场景描述> → 预期 <X> → 实际 <X> → ✅
- [x] 用例 2: <场景描述> → 预期 <X> → 实际 <X> → ✅
- [x] 用例 3(回归): <旧功能回归> → 预期 <X> → 实际 <X> → ✅

### 6.3 边界 / 异常
- [x] <边界场景,如空值/重复/越界> → ✅ 正确处理

### 6.4 自测结论
- 总体: ✅ 通过 / ❌ 不通过
- 遗留问题: <如无可写"无">
```

### 自测的硬性要求

- 自测**必须在向用户回报"完成"之前**完成,并写进本节
- 报告里要写**实际跑过的命令**和**真实输出**(可以贴关键几行),不能写"应该通过了"
- 自测失败 → 回到"任务中"继续修,不要在测试报告里写"已知问题"糊弄过去
- 用户指定的复现步骤 / 验收点必须出现在 `6.2 手工验证` 列表里,作为必过项

---

## 不要

- 在任务文件里写"完整代码" → 代码在代码仓库里
- 把多个不相关任务塞进同一份文件
- 只在 git commit 时才更新 → commit 是收口节点,不是维护触发点
- 跳过自测就标记任务"已完成"
- 同一对话内出现新需求时,懒得建新文档而把新需求塞进原文档 → 必须新开一份
- **任务涉及文件改动,但 `## 8. 改动的文件` 小节空着 / 只写"见 git commit" → 必须显式列文件路径**
