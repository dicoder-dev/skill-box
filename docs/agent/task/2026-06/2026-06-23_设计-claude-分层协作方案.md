## 4. 问题与方案

### 问题 1:`docs/project/` 已存在,不能与新方案的 `docs/agent/project/` 混淆

- **现象**: 用户需求里写了 `docs/agent/project`,但仓库里已经有 `docs/project/` 业务文档
- **定位**: 两套目录职责不同 —— `docs/project/` 是项目业务文档(给人看),
  `docs/agent/` 是 AI 协作上下文(给 Claude 看)
- **方案**: 在 `docs/agent/README.md` 与 `docs/agent/project/README.md` 里都明确
  写出"边界"小节,杜绝后续混淆
- **教训**: 设计分层前先盘点已有目录,避免命名撞车

### 问题 2:根 `CLAUDE.md` 体例约束

- **现象**: 容易把 CLAUDE.md 写成大百科,导致每次会话都吞大量上下文
- **定位**: Claude Code 默认会读根 `CLAUDE.md`,文件越大越浪费
- **方案**: 根 CLAUDE.md 只放"项目一句话 + 必读文档表 + 铁律 + 命令速查",
  不超过 100 行,详细全部按需 @ 到 `docs/`
- **教训**: 入口文件应当是"目录",不是"内容"
