## 4. 问题与方案

### 工具列表数据源
- 后端没有"list tools"接口,前端 scope-status 返回的 tools 数组依赖 current 选中(弹窗未打开时 current 为空)
- 决定:写死 5 个工具(Codex / Claude / OpenCode / Cursor / Trae),跟前端 TOOL_ICON_MAP / TOOL_DISPLAY 保持一致
- 跟 scope-status 工具行用的是同一份硬编码,语义对齐

### 弹窗内项目下拉数据
- 用现有 `listProjects` 客户端(`/api/skillbox/projects?page=1&size=200`)
- 弹窗打开时拉一次(不管 scope 选啥);后端分页 size 200 够用
- 默认选中第一项(draft.project_id=0 时)避免用户必选

### apply 失败的容错
- createSkill 失败 → 阻断,弹 error(原本逻辑)
- createSkill 成功 + apply 失败 → 不阻断,弹"部分失败"toast,告知哪些工具失败
- createSkill 成功 + apply 全部成功 → 弹"已在 N 个工具上自动启用"toast
- 用户体验:用户明确选了工具就期望生效,失败要让用户知道;但 createSkill 本身已经成功,不应回滚

### scope=project 但未选项目
- 加 `errProjectRequired` 校验,前端阻断
- 防止后端 project_id=0 + scope=project 的非法组合

### editing 模式(已有 skill)的行为
- scope / project_id / 适用工具编辑态全部 disabled
- 工具不重新 apply(避免老 skill 在多工具上重复创建文件)
- 提示用户去详情区用 scope chip 改生效位置
