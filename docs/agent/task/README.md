# docs/agent/task — 任务过程文件

> 每一个"有具体目标"的对话 / 子任务都应当在这里留一份过程文件。
> 不是给人看的进度汇报,是给 **未来的 Claude / 你自己** 回看的"我们当时怎么做的"。

## 何时建

**默认都要建。** 任何对话都值得留一份过程文件 —— 哪怕只是闲聊,也建一份简短记录,
保留当时的上下文、用户原话、Claude 答复,日后翻看时能记起"为什么这样答"。

- 用户给了一个具体任务(不是闲聊) → 完整过程文件
- 一两句话就能答完的闲聊 → 简短记录(需求 / 答复 / 决定)
- 纯信息查询("XX 文件在哪") → 可以省略

不建的话,几个月后回看会完全想不起来当时聊过什么。

## 目录结构(按月份分组)

```
docs/agent/task/
├── README.md               # 本文件
├── _template.md            # 任务文件模板
└── YYYY-MM/                # 按月份分子目录,一个月一个文件夹
    ├── MM-DD_<任务类型>_<主题>.md
    ├── MM-DD_任务类型<主题>.md
    └── ...
```

- **月份目录命名:`YYYY-MM/`**(例如 `2026-06/`)
- **文件名:`MM-DD_<任务类型>_<主题>.md`**
- 同月份的任务都进同一个 `YYYY-MM/` 子目录,避免 `task/` 根目录文件膨胀
- 新月份第一天如果发现还没有对应目录,先 `mkdir -p` 再放文件

## 文件命名

```
docs/agent/task/YYYY-MM/<任务类型>_<主题>.md
```

- **月份目录:`YYYY-MM/`**(例如 `2026-06/`),按对话开始的月份归类
- **文件名:`YYYY-MM-DD_<主题>.md`**,日期用对话开始的日期(本地时区)
- 主题:用动名词短语,简洁可搜索
  - `docs/agent/task/2026-06/06-23_功能开发_claude-分层协作方案.md` ✅
  - `docs/agent/task/2026-06/06-23_bug修复-sys_user-搜索-参数校验.md` ✅

## 文件结构

```markdown
# <任务主题>

**日期:** YYYY-MM-DD
**状态:** 进行中 | 已完成 | 已暂停

## 1. 需求
用户原始诉求 + 任何澄清后的细化目标。

## 2. 任务列表
- [ ] 子任务 1
- [ ] 子任务 2
- [x] 已完成子任务

## 3. 执行进度
时间倒序记:
- HH:MM 完成 XXX
- HH:MM 发现 XXX 问题
- HH:MM 决定 XXX

## 4. 问题与方案
> 开发中遇到的非平凡问题(>5 分钟定位或设计取舍)。
> 每个问题:现象 → 定位 → 方案 → 教训

## 5. 需求回流
> 用户临时加塞 / 计划外需求。完成后并入 `docs/project/需求规划.md`。

## 6. 总结(任务结束时填)
- 完成了什么
- 留下了什么(代码 / 文档 / 决策)
- 留给下次的事
- 复盘:哪里做得好,哪里能改进
```

## 维护规则

- **任务开始**:复制 `_template.md` 改名,放到当月目录 `YYYY-MM/`(没有就 mkdir),
  立即填"需求 / 任务列表"
- **任务中**:每次有进展,更新"任务列表"和"执行进度"
- **任务结束**:
  1. 把状态改为 `已完成`
  2. 填"总结"
  3. 决定是否要更新 `docs/project/` 业务文档或 `docs/agent/memory/` 长期记忆
- **不要**:
  - 在任务文件里写"完整代码" → 代码在代码仓库里
  - 把多个不相关任务塞进同一份文件
  - 任务结束后还继续追加内容(开新文件)
