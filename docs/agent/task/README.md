# docs/agent/task — 任务过程文件

> 每一个"有具体目标"的对话 / 子任务都应当在这里留一份过程文件。
> 不是给人看的进度汇报,是给 **未来的 Claude / 你自己** 回看的"我们当时怎么做的"。

## 何时建

- 用户给了一个具体任务(不是闲聊)
- 任务预计会跨多轮对话 / 多个子步骤
- 涉及代码改动 / 设计决策 / 踩坑

不需要建:
- 一两句话就能答完的闲聊
- 纯信息查询("XX 文件在哪")

## 文件命名

```
YYYY-MM-DD_<主题>.md
```

- 日期:任务 **开始** 的日期(本地时区)
- 主题:用动名词短语,简洁可搜索
  - `2026-06-23_设计-claude-分层协作方案.md` ✅
  - `2026-06-23_docs.md` ❌(太泛)
  - `2026-06-23_修复-sys_user-搜索-参数校验.md` ✅

## 文件结构(模板见 `_template.md`)

```markdown
# <任务主题>

**日期:** YYYY-MM-DD
**状态:** 进行中 | 已完成 | 已暂停

## 1. 需求
用户原始诉求 + 任何澄清后的细化目标。

## 2. 任务列表
- [ ] 子任务 1
- [ ] 子任务 2
- [x] 已完成子任务

## 3. 执行进度
时间倒序记:
- HH:MM 完成 XXX
- HH:MM 发现 XXX 问题
- HH:MM 决定 XXX

## 4. 问题与方案
> 开发中遇到的非平凡问题(>5 分钟定位或设计取舍)。
> 每个问题:现象 → 定位 → 方案 → 教训

## 5. 需求回流
> 用户临时加塞 / 计划外需求。完成后并入 `docs/project/需求规划.md`。

## 6. 总结(任务结束时填)
- 完成了什么
- 留下了什么(代码 / 文档 / 决策)
- 留给下次的事
- 复盘:哪里做得好,哪里能改进
```

## 维护规则

- **任务开始**:复制 `_template.md` 改名,立即填"需求 / 任务列表"
- **任务中**:每次有进展,更新"任务列表"和"执行进度"
- **遇到问题**:追加到"问题与方案",**不要写到 `feedback_*.md`**(那是通用规则)
- **任务结束**:
  1. 把状态改为 `已完成`
  2. 填"总结"
  3. 决定是否要更新 `docs/project/` 业务文档或 `docs/agent/memory/` 长期记忆
- **不要**:
  - 在任务文件里写"完整代码" → 代码在代码仓库里
  - 把多个不相关任务塞进同一份文件
  - 任务结束后还继续追加内容(开新文件)

## 与其他目录的关系

```
docs/agent/task/<本次>.md
  ├── 完成后学到的通用规则 → 提炼到 docs/agent/memory/feedback_*.md
  ├── 完成后发现的新的项目状态 → 提炼到 docs/agent/memory/project_*.md
  ├── 计划外需求 → 写"需求回流",再并入 docs/project/需求规划.md
  └── 非平凡的踩坑 → 提炼到 docs/project/开发报告.md
```

任务文件是 **临时文档**,提炼后才进入长期文档。
