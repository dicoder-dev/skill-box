# 2026-07-01 市场栏目 skill 搜索接口完善

## 1. 需求

靓仔反馈:**「市场」栏目显示的 skill 不全,且搜索功能没办法确定**。
根因:
1. **skillhub 源永远只兜底 3 条**:现有 `skillhub.Adapter` 走 `/api/v1/skills`(旧路径,实际不存在),任何调用都返空 → fallback 到 `knownFallback`(3 条静态条目)。
2. **搜索框只是后端 SQL LIKE**:前端 onSearch 把 `keyword` 拼到 `?keyword=xxx`,后端 `Orchestrator.ListSkills` 走 `market_skills` 表 `name LIKE '%kw%'`,依赖缓存数据。
3. **skillhub 真实 API 已抓到**:`docs/project/apis/skillhub-api.md` 含 23 个接口,核心 `GET https://api.skillhub.cn/api/skills?keyword=&sortBy=&order=&pageSize=` 无需鉴权。

## 2. 决定

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 适配策略 | skillhub 走真实 JSON API;skills.sh 保留 HTML | skills.sh 公共 API 字段缺 description/version/tags;HTML 解析仍是最全数据源 |
| 接口签名 | `MarketAdapter.Discover` 加 `keyword` 参数 | Detail/Download 签名不变;前端搜索框走三方源搜索语义 |
| 兜底 | skillhub 失败时仍走 knownFallback(3 条) | UI 不空白,降级已知;现状行为不变 |
| skillhub timeout | 20s → 30s | 真实 API `pageSize=100` 单页慢,需要更长 |
| skills.sh | 走 `/search?q=` 搜索页;失败时 substring 过滤 fallback | 保留 HTML 兜底 + 新增 keyword 透传 |

## 5. 问题与方案

### 5.1 skillhub 文档与实际响应字段名差异
- 现象:文档说 `tags: []`,实际响应可能 `tags: {}`(空 map)
- 方案:json.Unmarshal Go 自动处理,空 map 也能 unmarshal 为 nil map,`for _, t := range nil` 安全;已在 `dedupStrings` 容忍 nil 输入
- 教训:依赖 JSON unmarshal 的宽容性,不要 strict 解析;fallback 兜底永远留 3 条

### 5.2 zip 50MB cap 不够
- 现象:恶意大包可能撑爆内存
- 方案:`io.LimitReader(resp.Body, 50<<20)` 50MB cap,截断时走 fallback
- 教训:任何外部解压都要 cap,Go stdlib `archive/zip` 在 zip.NewReader 上不做大小校验

### 5.3 skills.sh `/search?q=` 路径不一定存在
- 现象:站点改版后路径可能失效
- 方案:404 时 fall through 到 `knownCatalogFallback` + substring 过滤,UI 不空白
- 教训:不要依赖三方站点的固定路径,所有外部 URL 都要有 fallback 兜底

### 5.4 `archive/zip` 解压时 frontmatter 默认 version 不可信
- 现象:ParseSkillMD 给默认 `0.1.0`,zip 路径里的 version(`skills/{slug}/{ver}/SKILL.md`)才可靠
- 方案:解析后用 zip 路径推断的 version 覆盖 frontmatter(2026-07-01 改:只要 zip 路径有 version 就覆盖,优先级高于 frontmatter)
- 教训:派生数据(zip 路径)往往比手动声明的 frontmatter 更可靠

### 5.5 NewWithClient 双 client 限制
- 现象:zip flow 走 `NewWithClient` 时,noRedirectClient 和 httpClient 共用同一个 fakeRT;follow redirect 时 fakeRT 又跑一遍,匹配不到 zipServer URL,返 404
- 方案:`NewWithClients(noRedirect, normal)` 拆开两层 client,zip flow 测试 noRedirect 走 fakeRT(返 302),normal 走 default transport(真访问 zipServer)
- 教训:复杂 HTTP 流程测试需要分阶段 mock;single mock 不够用

### 5.6 skillmarket_test 旧路径已不存在
- 现象:`TestSkillHubAdapter_E2E_DiscoverDownload` 测旧 `/api/v1/skills` 路径,新 API 改到 `/api/skills`
- 方案:测试改写,新路径 + 新响应结构;保留 E2E 走 Discover+Download 链路的语义
- 教训:接口契约改的时候,所有引用旧契约的测试都要同步改,不能"留 alias"
