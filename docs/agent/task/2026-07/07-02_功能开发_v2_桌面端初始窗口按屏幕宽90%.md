## 4. 问题与方案

### 问题 1:首次 80% 没有生效的根因

- 现象:用户 `wails3 dev` 后窗口还是 1280×800。
- 排查:读 `desktop/wails_app.go` 当前代码,发现 `resizePrimaryToScreenRatio` 调用放在了
  `startupAsync` 协程的 `time.Sleep(500ms)` **之前**。
- 根因:`Window.GetScreen()` 在 alpha.60 实现里走
  `InvokeSyncWithResultAndError(w.impl.getScreen)`,要求 Wails 主循环 ready 才能消费消息。
  协程一启动就调,但主循环还没起来,InvokeSync 卡在等回包(可能超时失败)。窗口
  留在 1280×800 兜底。
- 方案:把 resize 移到 `time.Sleep(500ms)` **之后**,确保主循环 ready 后再调。
- 为什么 user 那次没看到 resize 失败日志:之前的代码在 GetScreen 失败时只 log warning
  不 panic,加上 `time.Sleep` 之前 goroutine 里没 sleep,这条日志很可能被刷得很快被
  窗口出现的视觉信号掩盖。修复后 sleep 让 Wails 主循环 ready 概率更高,resize 真正生效。

### 问题 2:高度比例公式改造

- 现象:之前 `newH = newW × 0.625`(16:10 推导),用户在 4K 屏上觉得"窄+扁"。
- 决策:宽高各自按屏宽/屏高 × 比例(`newH = screenH × 0.9`),不再从宽度反推。
- minW/minH 同样分别按屏宽/屏高 × minPrimaryWidthRatio(0.6)。

### 问题 3:屏幕高度获取

- 现象:之前只读 `screen.Size.Width`,不读 `Height`;函数签名只接收 widthRatio。
- 决策:函数签名加 `heightRatio float64` 参数,内部 `screenH = screen.Size.Height`,
  兜底到 `screen.Bounds.Height`。
