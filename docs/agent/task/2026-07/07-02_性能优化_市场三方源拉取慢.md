# 2026-07-02 市场三方源拉取慢优化

## 1. 需求

靓仔反馈:进入市场界面拉取三方源数据很久,skillhub(国内服务器)不应该慢但偶尔 45s 超时,skills.sh 也偶尔超时。

## 2. 根因分析

进市场慢的 5 个根因(从大到小):

1. **skillhub.Discover 串行翻页** — `skillhub.go:194-254` `for page := 1; !stop; page++`,40000 条 ≈ 几十秒
2. **skillssh.discoverFromAuditsAPI 串行翻页** — `skillssh.go:218-271` 50 页 audits API 单页 ~300ms(海外跨太平洋) ≈ 15s
3. **http.Client 没复用** — skillhub/skillssh.New() 都 `&http.Client{Timeout: 30s}`,没显式 Transport 配置,默认 MaxIdleConnsPerHost=2,**每次翻页都 TLS 重连**
4. **没设 Accept-Encoding: gzip** — Go http.Client 默认不自动 gzip,大 JSON 不压缩
5. **(已排除)前端 onMounted 串行调用** — listSkills + listSourcesAggregated 是本地缓存,本身 <100ms,不慢

## 3. 决定

### A. 抽公共 HTTP 客户端(`pkg/httpx`)

- 新建 `api-server/pkg/httpx/client.go`
- 统一配 `Transport.MaxIdleConnsPerHost=10 / IdleConnTimeout=90s / TLSHandshakeTimeout=10s / ExpectContinueTimeout=1s`
- 自动 `Accept-Encoding: gzip` + 用 `gzip.NewReader` 解压
- 工厂函数 `NewDefaultClient(timeout)` 给 skillhub/skillssh 用
- 单一职责,不加 singleflight(本次不需要)

### B. skillhub 翻页改并行(worker pool 限并发=4)

- `page 1..N` 用 4 路 worker 并发拉
- 退出条件保持:①累计 ≥ totalHint ②added==0 ③本页<pageSize ④ctx 取消
- 合并去重逻辑保留(seen map 加锁)

### C. skillssh 翻页改并行(限并发=4)

- 同 skillhub
- 第一页失败仍直接放弃 JSON 路径(单页判降级)
- 后续页失败 break(已有逻辑)

### D. 不动 controller / service 层

- controller 的 90s ctx 不动
- 前端不动(接口协议不变)

### E. 测试

- skillhub 并行翻页:fakeRT 加 200ms 随机延迟,断言总耗时 < 串行一半
- skillssh 同上
- 既有测试用例不能挂

### F. 自测 + commit + push

- `go build ./...`
- `go test ./internal/skillmarket/...`
- curl 真实接口看耗时
- commit 推送

### 5.2 第二轮 (实施: 抽 httpx + 改 skillhub + 改 skillssh)

- **用户原话**: "开始"
- **本轮做了**:
  1. 建 `pkg/httpx`(NewClient + GetJSONWithUA + UA + Accept-Encoding: gzip + keep-alive Transport)
  2. skillhub.New / skillssh.New 切到 httpx.NewClient
  3. skillhub.NewWithClient: 给 noRedirect 强制覆盖 CheckRedirect(测试 mock 兜底)
  4. skillhub.fetchBody / skillssh.fetchBody: 走 httpx.GetJSONWithUA
  5. skillhub.Discover: 串行 → 4 路 worker pool(缓冲 4)+ pageCh/resultCh + stopCh(select 救场)+ 排序合并
  6. skillssh.discoverFromAuditsAPI: 同样并行化
- **本轮踩坑**:
  - 死锁 1:pageCh/resultCh 都是无缓冲 → 实际是串行(并发失效)。改 buffered=maxConcurrency。
  - 死锁 2:producer 阻塞 send 时 stop 信号无效 → 加 stopCh + select 三选一。
  - 死锁 3:stub 测试场景下 stop 设上后 producer 不退出 → 加 ctx.Done select。
  - 死锁 4:E2E 测试 stub 单页返 1 条,totalHint=5000,但并发下"single page < pageSize" 不能逐 page 判定 → 把退出条件简化为 pagesLoaded*pageSize >= totalHint。
  - 顺序:并发后 page 1-3 顺序乱 → 收集后 sort.Slice 按 page 升序,保证测试期待的"翻页顺序"。
- **本轮决定**:最终方案 — buffered channel + stopCh/ctx.Done select + sort + 简化退出条件
- **本轮工具**: `Bash go build` / `Bash go test -timeout 60s`
- **状态**:实施完成,跑测试有 1-2 次 timeout,通过加 stopCh select 修好
