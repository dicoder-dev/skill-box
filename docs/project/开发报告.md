### 2.1 `NormalizeName` 把已存在的 `-` 也吃掉了

- **现象**:`NormalizeName("Already-Snake")` 返回 `"alreadysnake"`,期望 `"already-snake"`;连带 `TestParseSkillMD_RealTraeFile` 把 trae 的 `find-skills` 解析成 `findskills`。
- **根因**:原 switch 把 `[a-z0-9]` 与"空白类"两组字符分别处理,`-` 既不在第一组也不在第二组,被静默吞掉。
- **修复**:把 `-` 加入"折叠为 `-`"那一组的等价字符(等同于空白),同时保持连续 `-` 折叠为单个的语义。
- **复盘**:单元测试覆盖到了"保留已存在分隔符"这一类规约,但实现和测试是并行写的,实现先简化、测试后细化;以后涉及"折叠 / 归一化"类函数,先列等价类(保留 / 折叠 / 丢弃)再写代码。
- **涉及文件**:
  - [frontmatter.go](/Volumes/MyDrive/Home/dicoder/projects/skill-box/api-server/internal/skilladapter/frontmatter.go)
  - [adapters_integration_test.go](/Volumes/MyDrive/Home/dicoder/projects/skill-box/api-server/internal/skilladapter/adapters_integration_test.go)

### 2.2 `ParseSkillMD` 把"无 frontmatter"也走 H1 兜底

### 2.4 `go test ./...` 在工程根目录 hang 住

- **现象**:在 `api-server/` 下跑 `go test ./...` 偶尔会卡住超过 5 分钟不返回。
- **临时绕路**:本次先用 `go test ./internal/skilladapter/... ./internal/skillimporter/... ./internal/skillstore/... ./internal/settings/... ./internal/gapi/controller/skillbox/...` 这种"白名单"子集,跑完再决定要不要再扩。

### 2.5 沙盒环境 go build/test 需 `GOCACHE=/tmp/go-cache`

- **现象**:默认 `GOCACHE` 在沙盒下 read-only,`go build` / `go test` 报 `failed to initialize build cache`。
- **修复**:命令前加 `GOCACHE=/tmp/go-cache`(已加入根 `Taskfile.yml` 与本会话常用的脚本片段)。
- **影响**:所有 `go build` / `go test` 都要走这条;desktop 端 `task build` / `task dev` 内部已自带,无需另处理。

### 2.6 `c.JSON(500, ginp.H{...})` 编译失败

- **现象**:Step 2 写 onboarding controller 时,顺手用了 `ginp.H{...}` 拼错误响应,`go build` 报 `undefined: ginp.H`。
- **根因**:本仓库的 `ginp` 包没定义 `H` 别名;`gin.H` 是 `gin-gonic/gin` 自带的 `map[string]any`。
- **修复**:把 `ginp.H` 全部替换成 `gin.H`,并在 import 列表加 `"github.com/gin-gonic/gin"`。`logger.Infof` 同样误用 → 改 `logger.Info`(后者是 printf 风格)。
- **防止复发**:本仓库的错误响应有现成封装 `c.Fail(msgs...)` / `c.SuccessData(data, msgs...)`,非必要不走 `c.JSON` 直拼;真要直拼就用 `gin.H`。

### 2.8 where.Extra 字段名拼错导致编译失败

- **现象**:Step 3 写 `project.Service.List` 时,以为 `where.Extra` 有 `Page/Size` 字段(沿用了其它 ORM 的记忆),编译报 `cannot use ... &where.Extra{...} as ...`。
- **根因**:本仓库 `where.Extra` 实际是 `PageNum / PageSize`;`where.New` 返回 `*whereManager`,要 `.Conditions()` 转 `[]*Condition`。
- **修复**:sed 把字段名改对,顺手用 `where.New(...).Conditions()` 替换直接的 `where.New(...)`。
- **复盘**:跨项目搬代码最容易出这种字段名差异;以后写新 service 时,先 `cat pkg/where/*.go` 确认 struct 名再写,不要靠"应该这样"的肌肉记忆。
- **涉及文件**:[service.go](/Volumes/MyDrive/Home/dicoder/projects/skill-box/api-server/internal/project/service.go)

### 2.9 Python `str.replace` 在多行匹配时把 `&&` 引入 Go 源码

- **现象**:用 `python3 -c` 批量改 service.go 的 `where.Extra{...}` 块,replace 之后生成了 `&&where.Extra{...}` 这样的非法 Go 语法,后续 `sed` 又把它修复成了 `&wherewhere.Extra`。
- **根因**:replace 的 `find` 串里漏了 `&`,只替换了换行后的 `where.Extra`,遗留下一个孤立的 `&`;后续 sed 又把 `&where` 重复追加。
- **修复**:直接用 `cat > file <<EOF` 整段重写文件,一次性消除歧义。
- **复盘**:任何"先 sed 再 sed 救火"的链式改动都该被禁止;以后涉及 multi-line 文本改动,优先整段重写而非多轮 replace。

### 2.11 chat stream 不走 BindParamsHandler

- **现象**:Step 5 写 SSE controller 时第一反应是套 `ginp.BindParamsHandler`,发现它会调 `ctx.Success()` 兜底,而 SSE 必须是 `text/event-stream` + 多次 flush,二者不兼容。
- **修复**:`chat_stream.a.go` 改成直接注册 `gin.HandlerFunc`(`func(c *gin.Context)`),自己 `ShouldBindJSON` 拿 body、自己 `WriteHeader(200)` + `Header().Set(...)` + 手动 `Flush`。
- **复盘**:任何"边写 body 边变 header"的端点(不只是 SSE,还有 chunked download)都不能用 `BindParamsHandler`;后续若有上传/下载类需求,走同一份"裸 handler" 模板。
- **涉及文件**:[chat_stream.a.go](/Volumes/MyDrive/Home/dicoder/projects/skill-box/api-server/internal/gapi/controller/skillbox/caiprovider/chat_stream.a.go)

### 2.12 前端 SSE 解析的 3 个坑

- **现象**:第一版 `parseSSE` 在 chunk 边界 + 多行 `data:` 上都漏帧,前端 chat 历史出现"半句 markdown"。
- **根因**:
  1. `ReadableStream` 一次 `read()` 不一定恰好给一帧,要 buf 累加并按 `\n\n` 切
  2. SSE 允许一个事件多行 `data:`(同前缀),需要拼接而非只取第一行
  3. 帧内还可能混 `: comment` 心跳行,直接跳过
- **修复**:buf 按 `\n\n` 切,frame 内逐行解析,只收 `data:` 开头的行,空行 / `:` 行跳过;`[DONE]` 单独识别触发 onDone。
- **复盘**:任何 stream 解析都不该假设 "一次 read = 一条 message",必须有 buf 边界处理。
- **涉及文件**:[ai.js](/Volumes/MyDrive/Home/dicoder/projects/skill-box/frontend/src/api/skillbox/ai.js)

### 2.19 `restoreFromSnapshot` 不会删"已经空了的目标目录"

- **现象**:`TestBatchApplier_AtomicRollback` 失败 — atomic 模式下第一个 skill 成功落到 `root/a/`,第二个 skill 失败 → 应回滚 `a` 目录,但实际 `a` 还在。
- **根因**:`applier.go` 里 `restoreFromSnapshot` 在删除 post_files 后调用 `removeEmptyParents(filepath.Dir(full), dir)`,其中 `dir` 是**目标目录本身**(`root/a`);`removeEmptyParents` 的循环条件是 `dir != stopAt`,所以 `stopAt = dir = root/a` 时循环**不进入**,空目录不会被删。
- **修复**:把 `stopAt` 改成 `filepath.Dir(dir)`(即 `root`),让循环能删到目标目录本身(只删空目录,非空会提前 break)。
- **复盘**:回滚路径涉及"删文件 → 可能清空父目录 → 不应越界删到 stopAt 之外",但 stopAt 的"边界"含义应是"从这里往上的父链不要继续删",不是"不要删 stopAt 本身"。这种"边界语义模糊"在文件清理类函数里常见,以后写带 stopAt 的 walk 函数时,先在注释里把"边界是否包含"写清,再写循环条件。
- **涉及文件**:
  - [applier.go](/Volumes/MyDrive/Home/dicoder/projects/skill-box/api-server/internal/skillapp/applier.go) — `restoreFromSnapshot` 内 `removeEmptyParents` 调用

### 2.22 DeleteTag 0 行影响不返错(2026-06-22)

### 2.24 LinesDiff 折叠 long context 的 `...` 占位(2026-06-22)

- **现象**:`TestLinesDiff_ContextCollapse` 第一次写时,长 unchanged 段(100 行)没有被折叠,行数还是 200+,output 拖很长。

### 2.25 `ginp.RouterAppend` 的 Handler 期望 `gin.HandlerFunc` 而非 `*ginp.ContextPlus` 签名(2026-06-22)

- **现象**:第 10 步写 cskillpkg 时第一反应是按 `conboarding` / `cskill` 风格把 controller 函数签成 `func(c *ginp.ContextPlus)`,build 报 `cannot use ExportPackage (value of type func(c *ginp.ContextPlus)) as gin.HandlerFunc value`。
- **根因**:`ginp.RouterAppend` 的 `RouterItem.Handler` 字段类型是 `gin.HandlerFunc`(即 `func(*gin.Context)`),而 `*ginp.ContextPlus` 是 `ginp` 包的 wrapper 类型,不能直接赋值。需要 `BindParamsHandler` 这层"参数绑定 + 调用"包装来把 `*ginp.ContextPlus` 签名转成 `gin.HandlerFunc`。
- **修复**:export / import / preview 三个端点都是"raw body / octet-stream / SSE"风格,需要自己控制 `ShouldBindJSON` / `io.ReadAll` / 写 header / flush,跟 §2.11 chat_stream 同款,不走 `BindParamsHandler`;签名改为 `func(c *gin.Context)`,就能直接挂到 `ginp.RouterAppend.Handler`。
- **复盘**:路由注册风格有 3 种,(a) `BindParamsHandler` 包装的 `func(*ginp.ContextPlus, *Req)` 适合普通 JSON;(b) 裸 `gin.HandlerFunc` 适合 raw body / SSE / 上传 / 下载;(c) 走 `ginp.RouterAppend` 直接挂中间件也有过。写新 controller 之前先确认自己属于哪一类,再选签名。

### 2.26 `RequestExportPackage` 与 `skillpkg.ExportRequest` 是两个不同的 struct(2026-06-22)

- **现象**:cskillpkg 的 export controller 写 `svc.BuildExport(req)` 时,build 报 `cannot use req (variable of struct type RequestExportPackage) as skillpkg.ExportRequest value`。

### 2.27 以为 `mauditlog.Model` 有 `List` / `CountAll` / `GroupCount` 方法(2026-06-22)

- **现象**:写 saudit.GetStats 时第一反应是 `s.model().CountAll()` + `s.model().GroupCount(field)`,build 报 undefined。
- **根因**:mauditlog.Model 是 Step 1 自动生成的 dbops 风格,只有 `Create / FindOne / FindOneById / FindList / Update / Delete / DeleteById / GetTotal` 这套通用 CRUD,没有"总数 / group by"这种统计方法。
