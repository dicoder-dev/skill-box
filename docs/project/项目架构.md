# Skill Box 项目结构与使用说明

> 双部署形态(桌面端 + Web 端)的全栈应用,基于 Go(Wails / Gin)+ Vue 3 + Pinia。
> 一份业务代码、两份运行形态,共享同一套 HTTP API、配置、数据库与业务路由。

---

## 2. 部署形态对比

| 维度         | Web 端(单进程)                   | 桌面端(Wails)                    |
| ------------ | --------------------------------- | -------------------------------- |
| 入口         | `api-server/cmd/web/main.go`      | `main.go`(根目录)                |
| 进程模型     | 一个 OS 进程 = HTTP + 静态资源    | 一个 OS 进程 = HTTP + Wails 窗口 |
| 前端承载     | `http.FileServer(embed.FS)`       | Webview 加载 `http://127.0.0.1:port/` |
| 端口         | `configs.server.port`(默认 8082) | 启动时随机分配,前端通过 Wails 绑定获取 |
| 鉴权         | 由 `system.need_auth` 决定       | 通常 `false`(单用户本地)         |
| 桌面能力     | 不可用                            | 通过 `window.go.*` 绑定          |
| baseURL      | 空字符串(同源)                    | `http://127.0.0.1:<port>`        |
| 编译命令     | `go build -o bin/web ./api-server/cmd/web` | `wails3 build` 或 `task build`  |
| 调试         | `vite dev` + gin 代理             | `wails3 dev`                     |

### 启动流程图

```
                 ┌──────────────────────┐
                 │  configs.yaml 加载   │
                 └─────────┬────────────┘
                           ▼
                 ┌──────────────────────┐
                 │  dbs.SetDbType(...)  │
                 └─────────┬────────────┘
                           ▼
                 ┌──────────────────────┐
                 │  StartGinLogger      │ ── 写 logs/<date>.log
                 └─────────┬────────────┘
                           ▼
                 ┌──────────────────────┐
                 │  StartDB + 迁移      │ ── mysql / pgsql / sqlite
                 └─────────┬────────────┘
                           ▼
                 ┌──────────────────────┐
                 │  StartTask 定时任务  │
                 └─────────┬────────────┘
                           ▼
                 ┌──────────────────────┐
                 │  装配 ServerOptions  │
                 └─────────┬────────────┘
                           ▼
        ┌──────────────────┴──────────────────┐
        ▼                                     ▼
   Web 端:RUN(Serve 阻塞)         桌面端:Boot 返回 *Backend
                                              ▼
                                     Wails App.Run()(主循环)
```

### 为什么 Web 端能单进程?

`api-server/cmd/web/main.go` 启动后,HTTP server 同时承担:
- 业务路由(`/api/*` 由 `internal/gapi/router.Register` 注册)
- 静态资源(`/`,`/assets/*`,`/static/*`,SPA fallback → `index.html`)
- 运行时注入(每个 `index.html` 响应里在 `</head>` 前注入 `<script>window.__APP_RUNTIME__=…</script>`)

桌面端走同一套 `bootstrap.Boot + bootstrap.Serve`,只是把同一个 `embed.FS` 喂给
HTTP server,再让 Wails Webview 加载它,**业务代码零差异**。

## 3. 配置系统

### 3.1 配置文件

`configs.yaml` 是所有部署形态的**唯一**配置入口。
- Web 端启动时用 `-config` 标志或 `CONFIG` 环境变量指定路径
- 桌面端启动时用 `-config` 标志指定路径(默认 `configs.yaml`)

### 3.2 配置结构体

`api-server/configs/*.go` 下的每个 struct 都通过 `init()` 调用
`cfg.ParseConfigStruct(...)` 把自己注册到 `cfg` 包,**yaml 字段 ↔ struct 字段**
通过 `configkey` 标签(无标签则按 struct 字段名)映射。

`default` 标签提供兜底值(配置缺失时生效)。

### 3.3 关键配置项

```yaml
db:
  use_type: sqlite                  # mysql / pgsql / sqlite
  sqlite:
    db_path: data.db
  mysql:                             # use_type=mysql 时使用
    ip: 127.0.0.1
    port: "3306"
    user: root
    pwd: "123456"
    db: ""
    dbname: ""

server:
  port: "8082"                       # 监听端口(Web 端用,桌面端随机)

system:
  app_name: dianji                   # 应用名(注入到 __APP_RUNTIME__)
  run_mode: web                      # web / desktop(影响前端 baseURL 策略)
  need_auth: true                    # 是否启用 JWT 鉴权
  user_center_url: http://localhost:8082   # JWKS 来源

email / tencent:                    # 邮件 + 腾讯云 COS(按需)
```

### 3.4 system.run_mode / system.need_auth 的双重作用

| 配置                 | 后端行为                                     | 前端行为                                                       |
| -------------------- | -------------------------------------------- | -------------------------------------------------------------- |
| `run_mode: desktop`  | 桌面端启动脚本可读此值做差异化配置           | `baseURL=http://127.0.0.1:<port>`,`needAuth` 默认 false        |
| `run_mode: web`      | Web 端单进程                                 | `baseURL=''`(同源),`needAuth` 默认 true                        |
| `need_auth: true`    | 启用 `AuthUserCenterMiddleware`             | 请求层从 `localStorage.token` 注入 Authorization;401 跳 /login |
| `need_auth: false`   | 中间件直接 `c.Next()` 放行,不查 token       | 不注入 token;401 不自动跳转                                    |

`system.run_mode` 主要给运维/部署脚本使用;**运行时** 真正决定前端鉴权行为的是
后端注入的 `__APP_RUNTIME__.needAuth`(由 `need_auth` 字段同步过来)。

## 4. 后端架构

### 4.1 启动流程 `api-server/cmd/bootstrap`

三个入口:

| 函数 | 用途 | 适用场景 |
| ---- | ---- | -------- |
| `Run(opts)`     | `Boot + Serve` 阻塞 | Web 端 / 传统 CLI(单进程 = 一切) |
| `Boot(opts)`    | 完成 cfg/DB/Task,返回 `*Backend`(server 未启) | 桌面端需要先 Boot 再起 Wails |
| `Serve(b)`      | 阻塞跑 gin server   | 桌面端在 goroutine 里跑 Serve,主线程跑 Wails |

`ServerOptions` 字段含义见 `api-server/cmd/bootstrap/server.go`:
- `Addr`     : 监听地址(`:8082` 或 `127.0.0.1:0`)
- `ViewGlob` : 模板加载(桌面端留空)
- `StaticDir` / `StaticFS` : 静态资源
- `FrontRootFS` : **单页前端根**,启用 SPA fallback
- `SPAFallback` : `*bool`,默认 true

### 4.2 HTTP 路由装配顺序(`server.go:New`)

```
gin.New
  ├─ Logger / Recovery
  ├─ mountStatic       ← /assets, /static
  ├─ View 模板(可选)
  ├─ router.Register   ← 业务 /api/*(从 routers_import.go 收集)
  └─ mountFrontRoot    ← / + NoRoute(SPA fallback,index.html 注入 __APP_RUNTIME__)
```

**关键**:业务路由先注册,前端路由 fallback 在后 —— 避免 `/api/*` 被前端
catch-all 抢走。

### 4.3 业务路由注册(`ginp.RouterItem`)

`internal/gapi/router/routers_import.go` 用 `blank import` 把所有 controller
包引入(每个包有 `init()` 调 `ginp.RouterAppend(...)`),启动时一次性注册。

```go
ginp.RouterAppend(ginp.RouterItem{
    Path:           "/api/sys_user/create",
    Handler:        ginp.BindParamsHandler(Create, entity.User{}),
    HttpType:       ginp.HttpPost,
    NeedLogin:      false,        // 是否走 JWT 中间件
    NeedPermission: false,        // 是否走权限校验
    PermissionName: "SysUse.create",
    Swagger:        &ginp.SwaggerInfo{...},
})
```

- `NeedLogin=true` 的路由,`AuthUserCenterMiddleware` 会用 JWKS 解析
  `Authorization: Bearer <token>`,并把 token 存到 `c.Get("jwt_user")`。
- `NeedPermission=true` 才会进一步比对 `PermissionName`。
- 当前默认未挂载权限中间件(见 `router.go` 注释),只挂 `CORSMiddleware`。

### 4.4 控制器 / 服务 / 数据层

```
controller/<x>/c<x>/   <x>.c.go  ← 控制器(解析参数、调 service、写响应)
                            <x>.r.go  ← 路由声明(本包 init 里 RouterAppend)
service/<x>/s<x>/      <x>.go    ← 业务逻辑 + Model()(从 dbops 拿)
entity/<x>.e.go                  ← GORM 实体
model/<x>/<x>.m.go               ← dbops 的 Model 包装(读/写库分离)
dto/comdto/                      ← 通用 DTO(分页/搜索/响应)
```

调用模板(以 `cuser.Search` 为例):

```go
func Search(c *ginp.ContextPlus, params *comdto.ReqSearch) {
    if where.Check(params.Wheres) != nil { c.Fail(...); return }
    list, total, err := suser.Model().FindList(params.Wheres, params.Extra)
    if err != nil { c.Fail(...); return }
    c.SuccessData(&comdto.RespSearch{List: list, Total: uint(total), ...})
}
```

`c.SuccessData(data)` 会自动用 `{code:1, msg:"ok", data}` 结构返回,前端请求层
会自动剥出 `data`。

### 4.5 通用查询条件 `pkg/where`

```go
wheres := where.Format(
    where.OptEqual("id", 1),
    where.OptLike("username", "admin"),
    where.OptGt("created_at", t),
)
list, total, _ := userModel.FindList(wheres, extra)
```

`OptXxx` 系列覆盖 `=` / `!=` / `>` / `>=` / `<` / `<=` / `LIKE` / `IN` / `BETWEEN` 等。
`where.Check(wheres)` 校验参数合法性。

### 4.6 通用 DAO `pkg/dbops`

`dbops.NewBaseDb` 提供 `FindOne` / `FindList` / `FindById` / `Create` / `Update` /
`Delete` 等通用方法,所有实体都可以走它,不用自己写 SQL。

读库 / 写库可分离:`tables.NewUser(wdb, rdb)`,主从切换不需要改业务代码。

### 4.7 JWT 鉴权

- 启动时 `AuthUserCenterMiddleware` 一次性 `jwk.Fetch` 拉 JWKS(来自
  `system.user_center_url + /.well-known/jwks.json`)到全局 `keySet`。
- 每次请求按当前路径找 `RouterItem`;`NeedLogin=false` 的路由直接放行。
- 桌面端 `need_auth=false` 时,**所有**路径直接放行,不查表、不解析 token。

## 5. 前端架构

### 5.1 启动顺序 `main.js`

```
1. 读 window.__APP_RUNTIME__      → 同步(getRuntime)
2. 探测平台(window.go 绑定)        → 同步(platform.isDesktop)
3. await resolveBaseURL()         → 决定 baseURL
4. 写 Pinia store(useAppStore)
5. enableDebug()?(Vite dev / ?debug=req)
6. 探测 /api/health(走完整请求层)
7. app.mount('#app')
```

### 5.2 Pinia store `src/store/app.js`

```ts
useAppStore = {
    state:   { runMode, needAuth, appName, isDesktop, baseURL }
    getters: { isWeb, isDesktopMode, authEnabled, deployLabel }
    actions: { setRuntime, setPlatform, setBaseURL }
}
```

业务组件统一:
```js
import { storeToRefs } from 'pinia'
import { useAppStore } from '@/store/app.js'
const store = useAppStore()
const { isDesktop, runMode, needAuth, appName, baseURL } = storeToRefs(store)
```

### 5.3 平台能力 `src/platform/index.js`

按 `window.go` 是否存在返回不同实现:

| 调用                   | Web 端(默认实现)              | 桌面端(走 Wails 绑定)            |
| ---------------------- | ----------------------------- | --------------------------------- |
| `platform.app.getVersion()`         | `'web'`                        | `window.go.app.AppService.GetVersion()` |
| `platform.app.getServerPort()`      | `0`                            | `window.go.app.AppService.GetServerPort()` |
| `platform.app.quit()`               | no-op                          | `AppService.Quit()`               |
| `platform.window.toggleAlwaysOnTop()` | no-op                        | `WindowService.ToggleAlwaysOnTop()` |
| `platform.platform.os()`            | `'web'`                        | `PlatformService.OS()`            |
| `platform.platform.clipboardText()` | `''`                           | `PlatformService.ClipboardText()` |
| `platform.platform.openExternal()`  | `window.open()`                | `PlatformService.OpenExternal()`  |

**业务代码永远只 import `@/platform`**,不直接读 `window.go.*`。

### 5.4 HTTP 请求层 `src/core/utils/requests/`

```
core/utils/requests/
├── errors.js          ApiError / HttpError / BusinessError / TimeoutError
├── client.js          底层 fetch 封装(AbortController 串接 timeout + signal)
├── interceptors.js    拦截器栈(洋葱模型) + 默认拦截器安装
├── http.js            业务层 API(http.get/post/put/delete)
└── index.js           桶导出 + installDefaultInterceptors()
```

#### 调用风格

```js
import { http, BusinessError } from '@/core/utils/requests'

// 简单 GET
const data = await http.get('/api/health')

// POST with body
const user = await http.post('/api/sys_user/login', { username, password })

// GET with query params
const { list, total } = await http.get('/api/sys_user/search', { page: 1, size: 5 })

// 业务码失败会抛 BusinessError(code !== 1)
try { await http.get('/api/foo') }
catch (e) {
    if (e instanceof BusinessError) showToast(e.msg)
    throw e
}
```

#### 拦截器行为(默认)

1. **request 注入 token**:`needAuth=true` 时从 `localStorage.token` 取,
   加 `Authorization: Bearer ...` 头。
2. **response 401 兜底**:`needAuth=true` 时清 token + 跳 `/login`;
   `needAuth=false` 时仅清 token。
3. **response 业务码剥离**:响应体有 `code` 或 `success` 字段时,`code === 1`
   返回 `data`,否则抛 `BusinessError`。`/api/health` 这类非标准结构放行。

#### 调试日志 `src/core/utils/debug.js`

```js
import { dlog, derr, enableDebug, isDebug } from '@/core/utils/debug'
```

- 默认关闭。开启方式:① Vite dev(`import.meta.env.DEV`)、② URL `?debug` /
  `?debug=req` / `?debug=1`、③ 浏览器控制台 `__skillBoxDebug(true)`。
- 关闭时 `dlog`/`derr` 是 no-op,业务侧可放心调用。
- 关闭后所有 `[req]` 日志全部停掉,生产环境不会有性能损耗。

### 5.5 SPA 入口与运行时注入

后端在每个 `index.html` 响应前会注入:

```html
<script>window.__APP_RUNTIME__={"runMode":"web","needAuth":true,"appName":"dianji"};</script>
```

注入点优先 `</head>`,次选 `<body`,兜底头部。前端 `getRuntime()` 同步读取,
**main.js 第一行就能拿到**,不需要等异步。

## 6. 快速上手

### 6.1 环境要求

- Go 1.25+(项目使用 `go 1.25` + go workspace)
- Node 18+ / npm
- Wails CLI v3(`go install github.com/wailsapp/wails/v3/cmd/wails3@latest`)
- 数据库:SQLite 默认开箱即用;MySQL / PostgreSQL 按需
- (可选) Taskfile:`go install github.com/go-task/task/v3/cmd/task@latest`

### 6.2 配置

`configs.yaml` 默认就是 SQLite + Web 端 + 启用鉴权,直接 `go run` 即可。
如需切换:
- 用 MySQL:把 `db.use_type` 改为 `mysql` + 填写连接信息
- 桌面端鉴权:把 `system.need_auth` 改为 `false`

### 6.3 启动 Web 端(开发模式)

```bash
# 1) 启动前端 dev server(Vite 监听前端热更新)
cd frontend
npm install
npm run dev                 # 默认 :5173,Vite 会代理 /api 到后端

# 2) 启动后端(另一个终端,根目录)
go run ./api-server/cmd/web -config configs.yaml
# 访问 http://localhost:8082/
```

> 也可以用 `wails3 task web` 一个终端并行起后端 + 前端,详见 §8。

### 6.4 启动 Web 端(单进程生产模式)

**手动版**(分两步):

```bash
cd frontend
npm run build               # 产物在 frontend/dist/

cd ..
go run ./api-server/cmd/web -config configs.yaml
# 访问 http://localhost:8082/
# 前端 dist 已经被 embed.FS 嵌入,无需 nginx
```

**Taskfile 版**(推荐,自动处理 dist 同步 + 增量 build):

```bash
wails3 task run:web
# 内部会跑:web:go:mod:tidy → web:build:frontend → web:sync:embed → web:build → 启动 bin/web
# 访问 http://localhost:8082/
```

完整任务清单与设计取舍见 §8。

### 6.5 启动桌面端

```bash
# 方式一:用 wails3 dev(热更新,推荐开发时)
wails3 dev -config ./build/config.yml -port 9245

# 方式二:用 Taskfile
task dev

# 方式三:编译发布版
task build                  # 默认本平台
# 产物在 bin/<app>-<platform>-<arch>/
```

桌面端启动后会:
1. 进程内起 HTTP server(端口随机)
2. 拉起 Wails 主窗口
3. Webview 加载 `http://127.0.0.1:<port>/`
4. 前端通过 `window.go.app.AppService.GetServerPort()` 拿到端口,自动走对 baseURL

### 6.6 添加一个新业务接口

完整 demo —— 给 sys_user 加一个 `/api/sys_user/list_by_ids` 接口:

**1) 后端**

ginp.RouterAppend(ginp.RouterItem{

type ReqListByIds struct {

**2) 前端**

## 7. 常见问题

### Q1: 桌面端 baseURL 怎么拿到的?
`src/api/http.js:resolveBaseURL()` 会按 `Wails 绑定 → 健康探测 → 当前 origin`
顺序找后端。拿到后缓存到模块作用域,后续所有请求复用。

### Q2: 改了前端代码,桌面端要重新编译吗?
- 开发期用 `wails3 dev` 会自动监听 `frontend/dist` 变化并重启
- 生产期需要 `cd frontend && npm run build` 后重新 `go build`

### Q3: 桌面端怎么禁用鉴权?
`configs.yaml` 把 `system.need_auth` 设为 `false`。后端中间件会直接放行,
前端请求层不会注入 token、401 也不会跳 /login。

### Q4: 业务路由怎么避免被 SPA fallback 抢走?
`router.Register(r)` 必须在 `mountFrontRoot` 之前调用(`server.go:New` 已经
保证)。业务路由用具体路径( `/api/sys_user/search`),与 SPA fallback 不会冲突。

### Q5: 怎么添加数据库类型?
1. `pkg/dbops/base_db_new.go` 添加驱动初始化
2. `internal/db/dbs` 同步 type
3. `configs/db.go` 加配置项,`use_type` 支持新值

### Q6: JWKS 拉取失败怎么办?
- `user_center_url` 必须能从本机访问
- 默认鉴权流程会因 `keySet == nil` 直接返回 500;把 `need_auth` 临时改 false
  可绕开,便于本地调试
- 自建用户中心的 JWKS 路径约定:`<user_center_url>/.well-known/jwks.json`

### Q7: 控制台没看到 [req] 日志?
- 默认是关的
- 加 `?debug=req` 或浏览器控制台执行 `__skillBoxDebug(true)`
- Vite dev 模式自动开

## 8. Web 端 Taskfile 任务链(本次新增)

之前 Web 端启动方式散落在两个终端:`cd frontend && npm run dev` + `go run`。
本次在根 `Taskfile.yml` 新增了一组 `web:*` 任务(去掉 `common:` 前缀,与桌面端
`build/package/run/dev` 平级),提供"一次命令搞定"和增量构建能力。

### 8.1 任务清单

| 任务 | 用途 |
|------|------|
| `web:go:mod:tidy` | 同步 `api-server/go.mod`,纯 Go 依赖 |
| `web:build:frontend` | 仅构建前端 `dist`,**不触发 wails3 绑定生成** |
| `web:sync:embed` | 把根 `frontend/dist` 同步到 `api-server/cmd/web/frontend/dist`(`//go:embed all:frontend/dist` 路径相对于 `cmd/web` 包) |
| `web:build` | 完整生产构建:Vite dist → embed 同步 → `go build`。配 `sources`/`generates`,源码未变会跳过整条链 |
| `run:web` | 跑 `bin/web`,`deps: [web:build]` 自动 build,无脑运行 |
| `web:dev:backend` | `go run ./api-server/cmd/web`,后端独立热重启 |
| `web:dev:frontend` | Vite dev server,端口受 `WAILS_VITE_PORT` 控制 |
| `web` | 单终端并行起后端 + 前端,Ctrl-C 一并停 |
| `web:clean` | 清 `bin/web` + embed 同步目录 |

### 8.2 关键设计取舍

- **不在 `web:build:frontend` 里调 wails3**:纯 Web 端不需要 wails 绑定,
  `web:build:frontend` 故意不依赖 `common:generate:bindings`。
  桌面端仍走 `build/Taskfile.yml` 的 `build:frontend`,互不污染。

- **embed 同步目录是必须的**:`api-server/cmd/web/main.go` 里写的是
  `//go:embed all:frontend/dist`,路径相对于该 .go 文件。根目录的
  `frontend/dist` 必须先复制过来(由 `web:sync:embed` 做),否则 go build
  阶段 `frontend/dist` 不存在会编译失败。

- **web:sync:embed 也带 sources/generates**:`frontend/dist/**/*` 变化才会
  重新同步,避免每次 build 都做 cp。

- **不依赖 trap 来收尾**:Task 调 cmds 时宿主 shell 是 dash 系,连 `2` 都报
  `invalid signal specification`,所以 `web` 任务用 `& + wait` 模式:
  后端后台跑 → 前端后台跑 → wait 前端 → kill 后端收尾。Ctrl-C 时整个进程组
  收到 SIGINT,前后端一起退。

- **Windows 上 `&` 不支持**:`web` 任务在 Windows 上跑不了。Windows 用户分两个
  终端跑 `wails3 task web:dev:backend` + `wails3 task web:dev:frontend`。

### 8.3 用法速查

```bash
# 1) 一次性 build + 跑(最常用,生产模式)
wails3 task run:web

# 2) 单终端并行开发(Vite 热更新 + 后端热重启)
wails3 task web

# 3) 分两个终端独立调试
wails3 task web:dev:backend   # 终端 A
wails3 task web:dev:frontend  # 终端 B

# 4) 改前端代码后只重建前端 + 重启
wails3 task web:clean && wails3 task run:web
```

### 8.4 顺手修的两个真 bug

让 Web 端能稳定运行,期间发现并修了两处老 bug,顺手记一下。

#### 8.4.1 configs 加载顺序问题(导致 `panic: MySQL 连接参数不能为空`)

`api-server/configs/*.go` 里的 `init()` 在程序启动时就跑 `cfg.ParseConfigStruct(...)`,
把 struct 字段填上 `default` 标签的值(如 `Db.UseType="mysql"`)。**这时候 viper
还没加载 `configs.yaml`**。`cfg.InitCfg()` 加载完 yaml 后会把 viper 实例换成
新加载的,但 struct 字段不会自动重读,导致 `Db.UseType` 永远是 tag 默认的 `"mysql"`。
`StartDB` 按 mysql 走 → `mysql.InitDb` 检查到 `ip/port/userName/dbName/dbPwd`
都为空 → panic。

**修复**:`api-server/cmd/bootstrap/bootstrap.go` 的 `Boot()` 里,
`cfg.InitCfg` 之后对 5 个 config struct 显式 reparse 一遍:

```go
cfg.ParseConfigStruct(configs.Db)
cfg.ParseConfigStruct(configs.Server)
cfg.ParseConfigStruct(configs.System)
cfg.ParseConfigStruct(configs.Email)
cfg.ParseConfigStruct(configs.Tencent)
```

这一步之前漏了,**意味着配置文件其实从来没生效过**,所有部署形态都按 tag
default 跑。修完之后 `db.use_type: sqlite` 才真正生效,Web 端不再 panic。

#### 8.4.2 `cfg.GetBool` 把 `"true"` 当成 false

`api-server/pkg/cfg/helper.go` 的 `GetBool` 原版:

```go
if val == "yes" || val == "ok" || val == "1" { return true }
if val == "no"  || val == "ng" || val == "0" { return false }
return false
```

yaml 写 `need_auth: true`,viper 转字符串是 `"true"`,不在白名单里 → 落到末尾
`return false` → `configs.System.NeedAuth` 永远是 false → `__APP_RUNTIME__.needAuth`
注入成 false → 前端 `authEnabled` getter 拿到 false → 请求层不注入 token,
401 不跳 `/login`。**Web 端的 JWT 鉴权整体被静默绕过**。

**修复**:换 switch 兼容 `true/false/yes/no/ok/ng/on/off/0/1`,大小写无关。

```go
func GetBool(k string) bool {
    checkInstance()
    val, _ := instance.GetString(k)
    switch strings.ToLower(strings.TrimSpace(val)) {
    case "true", "yes", "ok", "on", "1":
        return true
    case "false", "no", "ng", "off", "0":
        return false
    }
    return false
}
```
